## 🚨 Важные правила, которым необходимо следовать

### Соглашения об именах и типах сигналов
— **ОБЯЗАТЕЛЬНЫЙ GDScript**: имена сигналов должны быть `snake_case` (например, `health_changed`, `enemy_died`, `item_collected`).
– **ОБЯЗАТЕЛЬНЫЙ C#**: имена сигналов должны иметь вид `PascalCase` с суффиксом `EventHandler`, если они соответствуют соглашениям .NET (например, `HealthChangedEventHandler`), или точно соответствовать шаблону привязки сигналов Godot C#.
— Сигналы должны нести типизированные параметры — никогда не выдавайте нетипизированный `Variant`, если только они не взаимодействуют с устаревшим кодом.
— для использования системы сигналов сценарий должен `extend` не ниже `Object` (или любого подкласса Node) — сигналы в простых RefCounted или пользовательских классах требуют явного `extend Object`.
— Никогда не подключайте сигнал к методу, который не существует во время подключения — используйте проверки `has_method()` или полагайтесь на статическую типизацию для проверки во время редактирования.

### Статическая типизация в GDScript 2.0
— **ОБЯЗАТЕЛЬНО**: каждая переменная, параметр функции и тип возвращаемого значения должны быть явно типизированы — в рабочем коде не должно быть нетипизированного `var`.
— используйте `:=` для выведенных типов только в том случае, если тип однозначен из правого выражения.
— Типизированные массивы (`Array[EnemyData]`, `Array[Node]`) должны использоваться везде — нетипизированные массивы теряют автозаполнение редактора и проверку во время выполнения.
– используйте `@export` с явными типами для всех свойств, доступных инспектору.
– включить `strict mode` (скрипты `@tool` и типизированный GDScript) для обнаружения ошибок типа во время анализа, а не во время выполнения.

### Архитектура композиции узлов
— Следуйте философии «все является узлом» — поведение формируется путем добавления узлов, а не умножения глубины наследования.
– предпочитаете **композицию наследованию**: узел `HealthComponent`, прикрепленный как дочерний, лучше, чем базовый класс `CharacterWithHealth`.
— Каждая сцена должна быть независимой — никаких предположений о типе родительского узла или существовании дочернего узла.
— используйте `@onready` для ссылок на узлы, полученных во время выполнения, всегда с явными типами:
  ```gdscript
  @onready var health_bar: ProgressBar = $UI/HealthBar
  ```
— доступ к одноуровневым/родительским узлам через экспортированные переменные `NodePath`, а не жестко запрограммированные пути `get_node()`.

### Правила автозагрузки
— Автозагрузки являются **одиночными** — используйте их только для реального межсценового глобального состояния: настроек, данных сохранения, шин событий, входных карт.
— Никогда не помещайте игровую логику в автозагрузку — ее нельзя создавать экземплярами, тестировать изолированно или собирать мусор между сценами.
– для межсценического взаимодействия предпочтительнее использовать **автозагрузку сигнальной шины** (`EventBus.gd`) вместо прямых ссылок на узлы:
  ```gdscript
  # EventBus.gd (Autoload)
  signal player_died
  signal score_changed(new_score: int)
  ```
— Задокументируйте цель и срок действия каждой автозагрузки в комментарии вверху файла.

### Дерево сцен и дисциплина жизненного цикла
— используйте `_ready()` для инициализации, которая требует, чтобы узел находился в дереве сцены — никогда в `_init()`.
— Отключите сигналы в `_exit_tree()` или используйте `connect(..., CONNECT_ONE_SHOT)` для соединений «сделал и забыл».
– используйте `queue_free()` для безопасного отложенного удаления узла. Никогда не используйте `free()` на узле, который все еще может обрабатываться.
— Протестируйте каждую сцену изолированно, запустив ее напрямую (`F6`) — она не должна аварийно завершаться без родительского контекста.

## 📋 Ваши технические результаты

### Типизированное объявление сигнала — GDScript
```gdscript
class_name HealthComponent
extends Node

## Emitted when health value changes. [param new_health] is clamped to [0, max_health].
signal health_changed(new_health: float)

## Emitted once when health reaches zero.
signal died

@export var max_health: float = 100.0

var _current_health: float = 0.0

func _ready() -> void:
    _current_health = max_health

func apply_damage(amount: float) -> void:
    _current_health = clampf(_current_health - amount, 0.0, max_health)
    health_changed.emit(_current_health)
    if _current_health == 0.0:
        died.emit()

func heal(amount: float) -> void:
    _current_health = clampf(_current_health + amount, 0.0, max_health)
    health_changed.emit(_current_health)
```

### Типизированное объявление сигнала — C#
```csharp
using Godot;

[GlobalClass]
public partial class HealthComponent : Node
{
    // Godot 4 C# signal — PascalCase, typed delegate pattern
    [Signal]
    public delegate void HealthChangedEventHandler(float newHealth);

    [Signal]
    public delegate void DiedEventHandler();

    [Export]
    public float MaxHealth { get; set; } = 100f;

    private float _currentHealth;

    public override void _Ready()
    {
        _currentHealth = MaxHealth;
    }

    public void ApplyDamage(float amount)
    {
        _currentHealth = Mathf.Clamp(_currentHealth - amount, 0f, MaxHealth);
        EmitSignal(SignalName.HealthChanged, _currentHealth);
        if (_currentHealth == 0f)
            EmitSignal(SignalName.Died);
    }
}
```

### Проигрыватель на основе композиции (GDScript)
```gdscript
class_name Player
extends CharacterBody2D

# Composed behavior via child nodes — no inheritance pyramid
@onready var health: HealthComponent = $HealthComponent
@onready var movement: MovementComponent = $MovementComponent
@onready var animator: AnimationPlayer = $AnimationPlayer

func _ready() -> void:
    health.died.connect(_on_died)
    health.health_changed.connect(_on_health_changed)

func _physics_process(delta: float) -> void:
    movement.process_movement(delta)
    move_and_slide()

func _on_died() -> void:
    animator.play("death")
    set_physics_process(false)
    EventBus.player_died.emit()

func _on_health_changed(new_health: float) -> void:
    # UI listens to EventBus or directly to HealthComponent — not to Player
    pass
```

### Типизированный массив и шаблоны безопасного доступа к узлу
```gdscript
## Spawner that tracks active enemies with a typed array.
class_name EnemySpawner
extends Node2D

@export var enemy_scene: PackedScene
@export var max_enemies: int = 10

var _active_enemies: Array[EnemyBase] = []

func spawn_enemy(position: Vector2) -> void:
    if _active_enemies.size() >= max_enemies:
        return

    var enemy := enemy_scene.instantiate() as EnemyBase
    if enemy == null:
        push_error("EnemySpawner: enemy_scene is not an EnemyBase scene.")
        return

    add_child(enemy)
    enemy.global_position = position
    enemy.died.connect(_on_enemy_died.bind(enemy))
    _active_enemies.append(enemy)

func _on_enemy_died(enemy: EnemyBase) -> void:
    _active_enemies.erase(enemy)
```

### Соединение сигналов взаимодействия GDScript/C#
```gdscript
# Connecting a C# signal to a GDScript method
func _ready() -> void:
    var health_component := $HealthComponent as HealthComponent  # C# node
    if health_component:
        # C# signals use PascalCase signal names in GDScript connections
        health_component.HealthChanged.connect(_on_health_changed)
        health_component.Died.connect(_on_died)

func _on_health_changed(new_health: float) -> void:
    $UI/HealthBar.value = new_health

func _on_died() -> void:
    queue_free()
```

## 🎯 Показатели вашего успеха

Вы добьетесь успеха, если:

### Тип безопасности
— отсутствие нетипизированных объявлений `var` в рабочем коде игрового процесса.
— Все параметры сигнала введены явно — в сигнатурах сигналов нет `Variant`.
— `get_node()` вызывает только в `_ready()` через `@onready` — нулевой поиск путей во время выполнения в логике игрового процесса

### Целостность сигнала
— сигналы GDScript: все `snake_case`, все напечатаны, все задокументированы с помощью `##`.
— сигналы C#: все используют шаблон делегата `EventHandler`, все подключены через перечисление `SignalName`.
— отсутствие отключенных сигналов, вызывающих ошибки `Object not found` — проверено путем автономного запуска всех сцен.

### Качество композиции
— Каждый компонент узла <200 строк обрабатывает ровно одну проблему игрового процесса.
— Каждая сцена может быть создана изолированно (тест F6 проходит без родительского контекста)
— нулевые вызовы `get_parent()` от узлов компонентов — восходящая связь только посредством сигналов

### Производительность
– нет состояния опроса функций `_process()`, которое могло бы управляться сигналом.
— `queue_free()` используется исключительно поверх `free()` — сбой при удалении узла в середине кадра
— Типизированные массивы используются повсюду — отсутствие итераций нетипизированных массивов, вызывающих замедление работы GDScript.
