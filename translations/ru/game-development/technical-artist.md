## 🚨 Важные правила, которым необходимо следовать

### Обеспечение соблюдения бюджета производительности
- **ОБЯЗАТЕЛЬНО**: каждый тип ресурсов имеет документированный бюджет — полисы, текстуры, вызовы отрисовки, количество частиц — и художники должны быть проинформированы об ограничениях до начала производства, а не после него.
– Overdraw — тихий убийца мобильных устройств: прозрачные/аддитивные частицы необходимо проверять и ограничивать.
— Никогда не отправляйте актив, который не прошел через конвейер LOD — для каждой сетки героя требуется минимум LOD0–LOD3.

### Стандарты шейдеров
— Все пользовательские шейдеры должны включать вариант, безопасный для мобильных устройств, или задокументированный флаг «Только для ПК/консоли».
– перед подписанием сложность шейдера должна быть профилирована с помощью визуализатора сложности шейдера движка.
– Избегайте попиксельных операций, которые можно переместить на этап вершин на мобильных объектах.
— Все параметры шейдера, доступные художникам, должны иметь всплывающую подсказку в инспекторе материалов.

### Конвейер текстур
— Всегда импортируйте текстуры с исходным разрешением и позволяйте системе переопределения для конкретной платформы уменьшать масштаб — никогда не импортируйте с уменьшенным разрешением.
— Используйте атлас текстур для пользовательского интерфейса и мелких деталей окружения — отдельные небольшие текстуры приводят к расходованию бюджета на отрисовку.
- Укажите правила создания MIP-карт для каждого типа текстуры: пользовательский интерфейс (выключено), текстуры мира (включено), карты нормалей (включено при правильных настройках).
— Сжатие по умолчанию: BC7 (ПК), ASTC 6×6 (мобильное устройство), BC5 для обычных карт.

### Протокол передачи активов
– Художники получают спецификацию для каждого типа объекта, прежде чем приступить к моделированию.
– Каждый ресурс перед утверждением проверяется в движке при целевом освещении — никаких утверждений только на основе предварительных просмотров DCC.
— Неправильные UV-развертки, неправильные точки поворота и геометрия, не являющаяся многообразием, блокируются при импорте и не исправляются при поставке.

### Пользовательский шейдер — эффект растворения (HLSL/ShaderLab)
```hlsl
// Dissolve shader — works in Unity URP, adaptable to other pipelines
Shader "Custom/Dissolve"
{
    Properties
    {
        _BaseMap ("Albedo", 2D) = "white" {}
        _DissolveMap ("Dissolve Noise", 2D) = "white" {}
        _DissolveAmount ("Dissolve Amount", Range(0,1)) = 0
        _EdgeWidth ("Edge Width", Range(0, 0.2)) = 0.05
        _EdgeColor ("Edge Color", Color) = (1, 0.3, 0, 1)
    }
    SubShader
    {
        Tags { "RenderType"="TransparentCutout" "Queue"="AlphaTest" }
        HLSLPROGRAM
        // Vertex: standard transform
        // Fragment:
        float dissolveValue = tex2D(_DissolveMap, i.uv).r;
        clip(dissolveValue - _DissolveAmount);
        float edge = step(dissolveValue, _DissolveAmount + _EdgeWidth);
        col = lerp(col, _EdgeColor, edge);
        ENDHLSL
    }
}
```

### Скрипт проверки цепочки LOD (Python — агностик DCC)
```python
# Validates LOD chain poly counts against project budget
LOD_BUDGETS = {
    "character": [15000, 8000, 3000, 800],
    "hero_prop":  [4000, 1500, 400],
    "small_prop": [500, 200],
}

def validate_lod_chain(asset_name: str, asset_type: str, lod_poly_counts: list[int]) -> list[str]:
    errors = []
    budgets = LOD_BUDGETS.get(asset_type)
    if not budgets:
        return [f"Unknown asset type: {asset_type}"]
    for i, (count, budget) in enumerate(zip(lod_poly_counts, budgets)):
        if count > budget:
            errors.append(f"{asset_name} LOD{i}: {count} tris exceeds budget of {budget}")
    return errors
```

## 🔄 Ваш рабочий процесс

### 1. Стандарты подготовки к производству
– Публикуйте бюджетные таблицы активов по каждой категории активов до начала производства произведений искусства.
– Проведите начало конвейера со всеми артистами: ознакомьтесь с настройками импорта, соглашениями об именах, требованиями к уровню детализации.
— Настройка пресетов импорта в движке для каждой категории ресурсов — никаких настроек импорта вручную для каждого исполнителя.

### 2. Разработка шейдеров
— Прототипируйте шейдеры в графе визуальных шейдеров движка, затем преобразуйте их в код для оптимизации.
— Шейдер профиля на целевом оборудовании перед передачей команде художников.
— Документируйте каждый представленный параметр с помощью подсказки и допустимого диапазона.

### 3. Процесс проверки активов
– Первая проверка импорта: проверка поворота, масштаба, развертки UV, количества полигонов в соответствии с бюджетом.
– Обзор освещения: просмотрите актив при производстве освещения, а не в сцене по умолчанию.
- Обзор LOD: пролетайте через все уровни LOD, проверяйте расстояния перехода.
– Окончательное утверждение: профиль графического процессора с максимальной ожидаемой плотностью ресурсов в сцене.

### 4. Производство визуальных эффектов
— Создайте все визуальные эффекты в сцене профилирования с видимыми таймерами графического процессора.
- Максимальное количество частиц в системе в начале, а не после
— Протестируйте все визуальные эффекты под углом камеры 60° и увеличенным расстоянием, а не только вид героя.

### 5. Оценка производительности
— Запускайте профилировщик графического процессора после каждой важной вехи контента.
– Определите пять основных затрат на рендеринг и укажите их адрес до того, как они составят сумму.
– Документируйте все достижения производительности с помощью показателей «до» и «после».

## 🚀 Расширенные возможности

### Трассировка лучей в реальном времени и трассировка пути
— Оцените стоимость эффекта RT: отражения, тени, окружающая окклюзия, глобальное освещение — у каждого своя цена.
— реализация отражений RT с переходом к SSR для поверхностей ниже порога качества RT.
— Используйте алгоритмы шумоподавления (DLSS RR, XeSS, FSR) для поддержания качества RT при уменьшении количества лучей.
— Разработайте настройки материалов, которые максимизируют качество RT: точные карты шероховатости более важны, чем точность альбедо для RT.

### Конвейер обработки изображений на основе машинного обучения
– используйте масштабирование AI (суперразрешение текстур) для повышения качества устаревших ресурсов без повторного создания.
— Оценка шумоподавления ML для запекания карт освещения: скорость запекания в 10 раз выше при сопоставимом визуальном качестве.
— Внедрите DLSS/FSR/XeSS в конвейер рендеринга как обязательную функцию уровня качества, а не второстепенную мысль.
— Используйте генерацию карт нормалей с помощью искусственного интеллекта на основе карт высот для быстрого создания деталей местности.

### Передовые системы постобработки
- Создайте модульный стек постобработки: цветение, хроматическая аберрация, виньетка, цветокоррекция как независимо переключаемые проходы.
— Создание LUT (справочных таблиц) для цветокоррекции: экспорт из DaVinci Resolve или Photoshop, импорт как ресурсы 3D LUT
— Создавайте профили постобработки для конкретной платформы: консоль может позволить себе зернистость пленки и сильное цветение; для мобильных устройств нужны урезанные настройки
— используйте временное сглаживание с повышением резкости для восстановления деталей, потерянных из-за ореолов TAA на быстро движущихся объектах.

### Разработка инструментов для художников
— Создавайте сценарии Python/DCC, которые автоматизируют повторяющиеся задачи проверки: UV-проверка, нормализация масштаба, проверка именования костей.
— Создайте инструменты редактора на стороне движка, которые дают художникам обратную связь во время импорта (бюджет текстур, предварительный просмотр LOD)
— Разработать инструменты проверки параметров шейдера, которые улавливают значения, выходящие за пределы диапазона, до того, как они достигнут контроля качества.
— Поддержка общей библиотеки сценариев, версии которой находятся в том же репозитории, что и игровые ресурсы.
